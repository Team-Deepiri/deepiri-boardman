# Deepiri Boardman Status Snapshot

**Original snapshot:** 2026-05-08 (vs 2026-05-03 vision discussion).  
**Last updated:** 2026-05-15 — reflects team check-in (APIs/feature set largely complete) and recent LLM/Plaky/agent hardening.

## Executive Snapshot

- **Product/API:** Core Boardman behavior is in place: GitHub ↔ Plaky sync (issues, PRs, merge/QA flows), assignment endpoints, CLI, HTTP API, optional worker/queue, deploy runbook and scripts. Team view: *functionally there* for intended automation.
- **Agent / LLM:** Chat and multi-step tools work with local **Ollama** and cloud providers. Recent focus: reliable **tool calling** when models emit JSON in text, **Plaky v1/public** list/schema alignment (status options under `configuration.values`, list payload `fields` as arrays, group id shapes), **SSE** final replies, and **stricter** status matching (e.g. “In Progress” vs “Revisions In Progress”). Heuristic-first **PR↔task** and **GitHub↔Plaky identity** matching remain default; **LLM** rerank / identity resolver stays **opt-in** via env.
- **Scan:** Two distinct paths — (1) **`boardman scan` / `POST /api/v1/agent/scan`** — GitHub-sourced `DIRECTION.md`, commits, issues → structured JSON task proposals (and optional Plaky create when not dry-run). (2) **`scan_local_repo`** — agent **tool** over a **local directory** for bounded file/doc context; does not run the full scan pipeline unless the conversation drives it.
- **Gaps:** Scan *quality* still benefits from real `DIRECTION.md` content per repo and an eval rubric; agent latency/verbosity on small local models is a known UX issue (context length, follow-up turns).

## Current Risks / Gaps

- **`DIRECTION.md`** quality per repo still drives scan usefulness.
- **Local LLM:** truncation and slow multi-step loops can confuse UX unless users start fresh sessions or raise context limits where feasible (`OLLAMA_NUM_CTX`, etc.).
- **Observability:** webhook/PR-link failure reasons could use lightweight metrics/dashboards as before.
- **Release hygiene:** keep feature branches and status doc branch separate so doc snapshots stay reviewable without mixing product code.

## Recommended Next Steps

3. **Production defaults:** read-only or `allow_writes=false` for exposed agent surfaces unless explicitly enabled.
