# Boardman status snapshot

**Updated:** 2026-05-15 — repo-grounded summary of what is in place and where effort is leaning, aligned with the product vision (GitHub ↔ Plaky automation, AI scan, and board-manager agent).

## Executive snapshot

- **Automation core:** Issue and PR lifecycles are wired end-to-end: task mapping, PR linking (including heuristic candidate scoring with optional LLM rerank **off by default**), merge and QA-oriented status transitions, assignment and worker-facing endpoints.
- **APIs and CLI:** REST surface, webhooks, SQLite persistence, `boardman` commands for scan, agent chat, tasks, and deploy preflight/smoke helpers are in place; deployment docs and compose-based stacks are documented.
- **AI scan:** `boardman scan` / `POST /api/v1/agent/scan` turn **GitHub-sourced** context (`DIRECTION.md`, commits, issues, Plaky hints) into a **structured JSON task list** normalized for Plaky; quality still depends on real `DIRECTION.md` and eval discipline.
- **Agent chat:** Multi-step LangChain path with Plaky/GitHub/repo tools, `allow_writes` guardrail, session history, and streaming; **local Ollama** is supported with tooling fixes (textual tool-call coercion, schema-aware Plaky list filters, streaming/final-text alignment). Chat remains **model-sensitive** for latency and answer fidelity.
- **Identity and linking:** GitHub↔Plaky identity and PR↔task matching are **heuristic-first**; LLM assists exist but are **environment-gated** and not required for default behavior.

## Vision alignment

### 1) Understand what each repository is trying to achieve

**Partial**

- **Done:** `boardman init`, `boardman scan`, `boardman scan-all`, `boardman status`, registry/routing (`repos.yml`).
- **Done:** Scan pipeline in `boardman/services/scan_handler.py` (LLM prompt + JSON task normalization + optional Plaky create).
- **Gap:** Many repos still ship thin or template `DIRECTION.md`; until that content is real, “intent understanding” is only as good as GitHub metadata plus whatever the model infers.

### 2) Detect missing or emerging work from repo signals

**Partial–strong**

- **Done:** Webhook-aware PR/issue handling; heuristic PR↔task pipeline with scoring and triage when confidence is low; scan path for proposed tasks.
- **Gap:** No single agreed **eval set** (repos × rubric) proving consistent, non-duplicative, high-actionability task proposals.

### 3) Create and update Plaky tasks with clear routing

**Strong**

- **Done:** Task/subtask create/update, schema-aware field patching, board/group placement, QA picker and assignment routes.
- **Done:** Explicit separation of read vs write in the agent (`allow_writes`).

### 4) Keep GitHub and Plaky linked across the lifecycle

**Strong**

- **Done:** Issue↔task links, PR comments on tasks, merge-driven status updates, deferred completion when multiple PRs apply, review-driven QA transitions where schema allows.

## What is solid in the codebase (thematic)

| Area | Notes |
|------|--------|
| Deploy / ops | Runbook (`docs/DEPLOYMENT.md`), preflight and smoke scripts, worker vs API split documented. |
| Providers | Ollama, OpenRouter, and other LLM paths in settings/factory; agent and scan can target the configured provider. |
| PR ↔ Plaky | Deterministic scoring + recall; medium band can optionally call an LLM reranker (**disabled unless enabled in env**). |
| GitHub ↔ Plaky user | Scoring and ambiguity handling in identity match; optional LLM same-person pass **disabled by default**. |
| Agent / local model | Tool calls coerced when models emit JSON in content; Plaky public API shapes (e.g. status `configuration.values`, list payload `fields` lists) handled for list/filter tools; SSE path avoids empty replies when stream matches final state. |
| Local path context | **`scan_local_repo`** (agent tool) summarizes a **disk path** for the model; it is **not** the same pipeline as `boardman scan` (which is **GitHub + DIRECTION–centric**). |

## Gaps and risks

- **`DIRECTION.md` quality** is still the main limiter for scan usefulness; Boardman’s own direction file should model the bar you want org-wide.
- **Agent UX:** Read-first and write guardrails exist; richer **human confirmation** for bulk or risky mutations in product workflows is still a product decision.
- **Observability:** Operational confidence grows with repeated smoke runs, webhook replay practice, and light metrics on failure modes (Plaky patch rejects, link ambiguity).
- **Release hygiene:** Keep env examples and secrets out of commits; verify migrations and `.env` on each release branch.

---

## Near-term direction (recommended focus)

1. **Treat `DIRECTION.md` as product data** — concrete goals and non-goals per important repo, including Boardman.
2. **Scan quality eval** — small fixed set of repos, rubric (actionability, dedupe, alignment to direction), periodic reruns.
3. **Agent reliability** — continue tightening local-model behavior (context size, latency) without weakening heuristic-first automation paths.
4. **Production rehearsal** — one full path: webhook → issue → task → PR → comment/status → merge → terminal state.
