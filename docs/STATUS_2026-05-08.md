# Boardman status snapshot

**Updated:** 2026-05-15 — what the codebase delivers today and how it supports the product vision (GitHub ↔ Plaky automation, AI scan, and board-manager agent).

## Executive snapshot

- **Automation:** End-to-end issue and PR lifecycles — task mapping, PR linking with heuristic candidate scoring (optional LLM rerank available but **off by default**), merge- and QA-oriented status updates, assignment and worker-facing endpoints.
- **Surface area:** REST API, webhooks, SQLite persistence, full `boardman` CLI (scan, agent chat, tasks), plus deploy preflight/smoke scripts and documented Compose stacks.
- **AI scan:** `boardman scan` and `POST /api/v1/agent/scan` combine **GitHub context** (`DIRECTION.md`, recent commits, open issues, Plaky hints) into a **normalized JSON task list** ready for Plaky — structured titles, descriptions, priorities, optional field maps, and evidence-style metadata in the prompt contract.
- **Agent chat:** Multi-step LangChain loop with Plaky, GitHub, and repo tools; `allow_writes` guardrail; session history and streaming. **Local Ollama** is a first-class path, including handling for models that emit tool JSON in plain text, schema-aware Plaky reads (public API field shapes), and streaming aligned with final assistant text.
- **Matching philosophy:** GitHub↔Plaky identity and PR↔task matching are built on **explicit heuristics and scoring**; LLM assists exist for teams that want them and are **opt-in via configuration**.

---

## How this maps to the vision

### 1) Repository intent

Boardman already **pulls intent from the repo as live data**: `boardman init`, `boardman scan`, `boardman scan-all`, `boardman status`, and `repos.yml` routing feed a scan pipeline (`scan_handler.py`) that grounds the LLM in **DIRECTION.md plus GitHub activity**. Richer `DIRECTION.md` content in each repo **amplifies** what the same pipeline can express — the machinery is there to use it.

### 2) Emerging work from signals

Webhook-driven PR and issue flows, a **scored PR↔task** candidate path with triage when matches are ambiguous, and the **scan** flow give multiple ways to turn repo activity into actionable Plaky work. The stack is ready for **formal quality benchmarks** as an enhancement layer if the team wants standardized rubrics across repos.

### 3) Task creation and routing

Full **create / update / subtask** paths, **schema-aware** field patching, board and group placement, QA picker routes, and an agent that **defaults to read-only** unless `allow_writes` is enabled — so automation stays **deliberate** about mutations.

### 4) Lifecycle linkage

Issue↔task links, PR comments on tasks, **merge-driven** status updates, **deferred completion** when multiple PRs touch the same work, and review-driven QA transitions **where the board schema supports them** — the lifecycle story is implemented in code and configurable per board.

## What is implemented (by area)

| Area | What it gives you |
|------|-------------------|
| **Deploy / ops** | Runbook (`docs/DEPLOYMENT.md`), preflight and smoke scripts, clear API vs worker responsibilities. |
| **LLM providers** | Factory and settings support Ollama, OpenRouter, and other paths; agent and scan run on the configured provider. |
| **PR ↔ Plaky** | Recall + composite scoring; optional LLM rerank in the medium-confidence band for teams that enable it. |
| **GitHub ↔ Plaky users** | Identity scoring with ambiguity handling; optional LLM same-person assist for gray zones when enabled. |
| **Agent + local models** | Coercion of textual tool calls into structured tool steps; Plaky list/schema handling aligned with v1/public payloads; SSE behavior that still delivers a complete reply when stream and graph output coincide. |
| **Two “scan” shapes** | **`boardman scan`** / **`/agent/scan`**: GitHub + DIRECTION–centric, structured task JSON. **`scan_local_repo`** (agent tool): bounded filesystem summary for a path on disk, ideal for pairing with the model inside chat. |

## Potential enhancements (optional next steps)

- **Richer `DIRECTION.md` everywhere** — same scan pipeline, stronger inputs; Boardman can dogfood a detailed direction file.
- **Scan evaluation playbook** — repeatable repo set + rubric (actionability, dedupe, direction alignment) to track improvements over time.
- **Agent polish** — further tuning of context windows and latency for local models; UI affordances for confirmation on large writes if product wants more friction.
- **Operations** — more frequent smoke rehearsals, webhook replay habits, and lightweight dashboards on Plaky/API outcomes if the team wants deeper visibility.
- **Release hygiene** — keep continuing current practice: env examples without secrets, migrations verified on release branches.
